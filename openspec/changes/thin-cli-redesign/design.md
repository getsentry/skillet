# Design — Thin CLI Redesign

## Context

Skillet v0.x is ~11k LOC of TypeScript whose center of gravity is doing generation itself: an interactive spec-author, an orchestrator over four bundled prompt-agents, a vitest-subprocess eval runner, and provider credential plumbing. Benchmarks never demonstrated statistically significant lift, and the commit history shows the machinery being wrestled into stability rather than compounding value. OpenSpec (59k stars, MIT) proves the inverse architecture: a zero-LLM CLI that owns artifacts, grammar, validation, and state, while the user's existing coding agent supplies all intelligence.

## Goals / Non-Goals

**Goals**
- Skillet becomes a file/state manager + validator + mechanical eval runner. Zero LLM calls.
- Intent is codified in a reviewable markdown spec (`spec.md`) whose scenarios map to eval cases.
- Evals are cheap to author (YAML), honest to interpret (trials + baseline lift), and run through real agents.
- One workflow surface for any host agent, generated by `skillet init --tools`, with all guidance served at runtime by the CLI.

**Non-Goals**
- Semantic skill grading inside the CLI; correctness signal comes only from evals.
- Backwards compatibility with `spec.yaml` / `*.eval.ts` at the CLI level (migration is an agent workflow).
- Multi-repo/team features, orchestration, or anything resembling a service.

## Decisions

### D1. Steal OpenSpec's architecture, not its domain

OpenSpec manages *change* artifacts for codebases; skillet manages *skill* artifacts. What we adopt is the architectural pattern: thin generated prompt files + fat CLI (`status`/`instructions --json` as the agent contract), filesystem-as-state-machine, a tiny strict markdown grammar with a mechanical validator whose errors carry fix hints, and multi-tool integration generation. What we do not adopt: delta specs and archives (a skill's spec is small enough to edit in place — git history is the change log), workflow schemas/DAGs (skillet has one fixed artifact chain: spec → skill+evals), and the changes/ directory model.

*Alternative considered*: building skillet **on top of** OpenSpec (skills as OpenSpec capabilities). Rejected: skills need a different grammar (triggers/constraints/behaviors vs requirements), a different lifecycle (living artifact, no archive), and an eval runner OpenSpec will never own. Depending on a fast-moving 1.x CLI for our core data model adds risk without removing much code.

### D2. spec.md replaces spec.yaml

Markdown wins over YAML for the intent artifact because humans review intent in PRs, agents already write structured markdown reliably, and OpenSpec demonstrates a heading grammar is mechanically parseable and validatable. The grammar keeps skillet-specific concepts that earned their place — triggers (SHOULD/SHOULD NOT) and constraints (MUST NOT) — and drops ones that didn't: `class` depth-gates, `dimensions`, `leakage_risk`, the 13-op SpecPatch machinery (agents edit markdown directly; `validate` catches breakage).

### D3. Evals spawn real agent CLIs (pluggable harness)

The v0.x built-in agent tested skills inside a synthetic pi-agent loop — not the environment skills actually run in. The new harness spawns actual coding agents (`codex exec` default, `claude -p` built-in) in a fresh workspace with the skill installed, which is both simpler (no agent loop, no provider code) and higher fidelity. A command template (`{workspace}`, `{prompt}`) makes any CLI a harness. Trade-off accepted: eval results depend on the harness agent's own variance and version — mitigated by `--trials` and by reporting the harness identity in results.

### D4. Judges run through the harness too

Keeping any direct LLM call would drag back provider autodiscovery, model selection, and API-key management. Instead a `judge:` check invokes the same harness CLI with a grading prompt and parses a pass/fail verdict. Binary verdicts replace 0–1 scores + thresholds — threshold calibration was pure churn in v0.x; repeatability now comes from trials. Risk: agent CLIs are chattier than raw model calls and verdict parsing can fail — mitigated by a strict "last line must be VERDICT: pass|fail" protocol, one retry, then `errored` (never silently failed).

### D5. Baseline lift is the headline metric

The original tool couldn't answer "does this skill actually help?" `--baseline` runs each case with and without the skill and reports per-behavior lift. This is the statsig story: same prompts, same harness, paired trials. It subsumes the old `compare` command.

### D6. Rewrite in place, greenfield src/

Keep the repo, npm name, and OpenSpec history; replace `src/` wholesale rather than incrementally refactoring — the deleted subsystems (authoring, agents, agent, eval, evals, staging, verify) are the majority of the code and the user is explicitly not attached to it. Salvage candidates: spec slug logic, SKILL.md frontmatter parsing, fixture-copy code, and the distilled writing guidance inside `agents/*/references/` (compressed into instructions payloads). New code gets real unit tests (parser, validator, harness templating, check evaluation) — feasible for the first time because nothing needs an LLM.

## Risks / Trade-offs

- [Harness variance makes evals flaky] → trials + pass rates instead of single-shot pass/fail; fail-fast when harness binary missing; harness identity recorded in results.
- [codex/claude CLI flags drift] → adapters are tiny (one command template each) and covered by unit tests on template substitution, not live calls.
- [Losing the interactive spec interview] → the propose workflow instructs the host agent to interview the user in its own conversation — a better UX than skillet's pause/resume ever was.
- [Instructions payloads bloat like the old prompt files] → hard budget: each artifact's instructions ≤200 lines; guidance beyond that must be cut, not appended.
- [Skill installation differs per harness] → adapter-specific install paths (e.g. workspace `.codex/skills/` vs `.claude/skills/`), covered by the harness spec's installation requirement.

## Migration Plan

1. Land the new CLI behind the same package name (major version bump).
2. `skillet status` detects legacy `spec.yaml` skills and directs to `/skillet:migrate`.
3. The migrate workflow (agent-driven) converts spec.yaml fields → spec.md sections and re-expresses eval intent as YAML cases; `.eval.ts` files are left in place for reference and ignored by the CLI.
4. Archive the three stale active OpenSpec changes without sync; this change's deltas update the living specs at its own archive time.

## Open Questions

- Should `--baseline` compute a significance test (e.g. exact binomial) or just report raw lift? Start with raw lift + trial counts; add stats only if users misread noise as signal.
- Does `skillet init` also scaffold a repo-level `skills/` convention, or stay per-skill-directory only? Deferred until real usage shows a need.
- Codex CLI non-interactive skill discovery: verify the exact mechanism (`--skip-git-repo-check`, skill dir location) during implementation; the harness adapter isolates whatever it turns out to be.
