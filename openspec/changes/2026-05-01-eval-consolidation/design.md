# Design

## Flow (the whole thing, end to end)

The user's complaint is that the flow isn't visible from the
output. Spelled out so the rationale for the consolidation stage
is clear:

```
spec.yaml ───────────────────────┐
                                 │
                                 ▼
              ┌──────────────────────────────────┐
              │         per-entry fan-out        │  parallel via AI queue
              │   (one job per behavior/must_not)│
              │                                  │
              │   ┌───────────────────────────┐  │
              │   │ 1. Generate plan (LLM)    │  │
              │   │   parse-retry up to 3x    │  │
              │   └─────────────┬─────────────┘  │
              │                 │ AssertionPlan  │
              │                 ▼                │
              │   ┌───────────────────────────┐  │
              │   │ 2. Verify plan (LLM)      │  │
              │   │    apply edits if needed  │  │
              │   └─────────────┬─────────────┘  │
              │                 │ EditedPlan     │
              └─────────────────┼────────────────┘
                                ▼
                    [all 29 plans collected]
                                │
                                ▼
              ┌──────────────────────────────────┐
              │     3. Consolidate (no LLM)      │  serial, in-process
              │                                  │
              │   - dedupe judges by name        │
              │   - extract fixture trees        │
              │   - rewrite plans to reference   │
              │     shared judges + fixtures     │
              └─────────────────┬────────────────┘
                                ▼
              ┌──────────────────────────────────┐
              │     4. Render artifacts          │  serial, in-process
              │                                  │
              │   - write evals/_judges.ts (1×)  │
              │   - write evals/fixtures/<slug>/ │
              │     for each case                │
              │   - write evals/<id>.eval.ts     │
              │     for each entry               │
              └──────────────────────────────────┘
```

Stages 1+2 are LLM-bound and parallel (one queue job per entry,
verify call piggy-backs inside the same slot). Stages 3+4 are
deterministic and run once after all entries have plans.

The contract between stages:

- **Stage 1→2**: a parse-validated `AssertionPlan` per entry.
- **Stage 2→3**: an `EditedPlan` per entry (verifier edits
  applied, or the original if edits were invalid). Same
  `AssertionPlan` shape; the verifier's polish is internal.
- **Stage 3→4**: a `ConsolidatedPlan` per entry, where each
  `judge` assertion still names a judge by name but the
  declarations are no longer in `plan.judges` — they live in
  the suite-level `_judges.ts`. `CasePlan.fixture` is a file map
  pointing at content the renderer will write to disk.
- **Stage 4 output**: `evals/_judges.ts`, `evals/fixtures/...`,
  `evals/<entry-id>.eval.ts` (one per entry).

## Per-skill output layout

```
skills/<skill>/
  evals/
    _judges.ts                          ← generated (1× per skill)
    fixtures/                           ← generated (per case)
      <case-slug-A>/
        .github/workflows/ci.yml
        scripts/run.sh
      <case-slug-B>/
        ...
    <behavior-id-1>.eval.ts             ← generated (1 per entry)
    <behavior-id-2>.eval.ts
    ...
```

## `_judges.ts` shape

```ts
// ──────────────────────────────────────────────────────────
// Generated by skillet eval-gen consolidation. Do NOT edit
// directly — judges are deduped from per-behavior plans on
// every regen. Edit behaviors via spec.yaml; their judge
// criteria are hand-editable in the regenerated plans before
// they land here, but direct edits to this file are clobbered.
// ──────────────────────────────────────────────────────────
import { judge } from "@sentry/skillet/evals";

export const IdentifiesPrivilegedTriggerJudge = judge(
  "IdentifiesPrivilegedTriggerJudge",
  async ({ criterion }) =>
    criterion("Names pull_request_target / workflow_run as the privileged trigger."),
);

export const RatesHighOrCriticalSeverityJudge = judge(
  "RatesHighOrCriticalSeverityJudge",
  async ({ criterion }) =>
    criterion("Treats the finding as HIGH or CRITICAL severity, not minor hardening."),
);

// ... one declaration per canonical judge in the suite
```

The generated `.eval.ts` files import from `./_judges.js`:

```ts
import {
  IdentifiesPrivilegedTriggerJudge,
  IdentifiesPRControlledCheckoutJudge,
  ConnectsSecretsOrWriteTokenJudge,
  RatesHighOrCriticalSeverityJudge,
} from "./_judges.js";
```

## Fixture trees

Fixtures move from inline shell strings to real files on disk.
The plan's `fixture` field (replacing `setup`) is a record:

```ts
fixture: {
  ".github/workflows/ci.yml": "name: CI\non:\n  pull_request_target:\n  ...",
  ".github/actions/setup/action.yml": "name: setup\nruns:\n  using: composite\n  ...",
  "scripts/run.sh": "#!/bin/bash\nset -e\n...",
}
```

At consolidation time, skillet writes each entry of the map to
`evals/fixtures/<case-slug>/<relative-path>`, creating parent
directories as needed.

The generated eval calls `await harness.useFixture(<case-slug>)`,
which:

1. Resolves `<skill-root>/evals/fixtures/<case-slug>/`.
2. Recursively copies the tree into the per-test workspace
   (preserving directory structure).
3. Returns control to the test, which then calls `run(...)`.

The `harness.useFixture` method is added to the
`FixtureHarness<TInput>` type in `vitest-evals/types.ts`.

## Dedup heuristic

Initial implementation: **exact name match**. Two judges with the
same `name` are treated as the same canonical judge. The
canonical declaration is the first one encountered (insertion
order across the per-entry plans).

When two plans declare a judge with the same name but **different
criteria**, we keep the first criterion and log a `warn` event
naming the entries and the conflicting criteria so the user can
audit. This is rare in practice — the generator's prompt steers
toward stable naming, and judges with distinct semantics get
distinct names.

Fuzzy / criterion-similarity dedup is deferred. Exact-name match
catches the high-volume duplication; the residual long tail is
acceptable.

## CasePlan changes

```ts
export interface CasePlan {
  name: string;                       // also used as fixture slug
  tests_behavior: string;
  input: string;
  /** File map: relative workspace path → file content. Replaces `setup`. */
  fixture?: Record<string, string>;
  /** @deprecated — accepted for backwards-compat; converted to fixture
   *  via a single shell heredoc fallback if both are absent. */
  setup?: string;
  timeout?: number;
  assertions: Assertion[];
}
```

The generator emits `fixture` going forward. `setup` is still
parsed and run-shell-on-demand for plans authored against the
old contract; it's never produced by new generation.

## ConsolidatedPlan internal shape

```ts
interface ConsolidatedPlan {
  // Same as CasePlan except `fixture` is a slug (case name)
  // pointing at the on-disk fixture tree, not a content map.
  cases: Array<Omit<CasePlan, "fixture" | "setup"> & { fixtureSlug?: string }>;
  // No `judges` here — they're consolidated at suite level.
}
```

The renderer takes a `ConsolidatedPlan` plus the suite-level
canonical judge set and produces the final TS, importing only the
judges referenced by this entry's cases.

## New `eval-gen-consolidate.ts` module

```ts
export interface ConsolidationResult {
  /** Canonical deduped judges, sorted by name for stable output. */
  judges: JudgePlan[];
  /** Per-entry consolidated plan. Same order as input. */
  perEntry: Array<{ entryId: string; plan: ConsolidatedPlan }>;
  /** Fixtures to write: { caseSlug: { relPath: content } }. */
  fixtures: Record<string, Record<string, string>>;
  /** Conflicts found during dedup, surfaced for telemetry. */
  conflicts: Array<{ judgeName: string; criteria: string[] }>;
}

export const consolidate = (
  plans: Array<{ entryId: string; plan: AssertionPlan }>,
): ConsolidationResult;
```

Pure function. Unit-testable in isolation.

## Renderer changes

The existing `renderEvalFile(entryId, plan)` becomes
`renderEvalFile(entryId, plan, sharedJudges)` where
`sharedJudges` is the canonical set. The renderer:

- Walks the plan's `judge` assertions to determine which judge
  names this entry references.
- Emits `import { ... } from "./_judges.js"` for those names
  (sorted).
- Drops the in-file `const FooJudge = judge(...)` declarations
  (they live in `_judges.ts` now).
- For each case, if `fixtureSlug` is set, emits
  `await harness.useFixture(<slug>)` instead of the legacy
  `await harness.setup(<script>)`.

A new `renderJudgesFile(judges: JudgePlan[]): string` produces
the contents of `_judges.ts`.

## Generator/verifier prompt updates

Generator prompt:
- Replaces the `setup: string` field guidance with `fixture` —
  a file map, with examples showing the same content as a real
  multi-file fixture.
- Adds a "stable judge naming" section: judges should follow a
  consistent naming convention so cross-behavior dedup works.
  Recommended verb-prefix patterns: `Identifies…Judge`,
  `Rates…Judge`, `Connects…Judge`, `RecognizesNo…Judge` (for
  must_nots).
- Notes that judges declared by this entry may be deduped
  against judges declared by other entries — if you write
  "IdentifiesPrivilegedTriggerJudge" here, it'll consolidate
  with the same-named judge from another behavior. Don't worry
  about that explicitly; just use stable names.

Verifier prompt:
- Same caps as before (≤5 judges per file is now ≤5 per entry's
  declared set; the consolidated `_judges.ts` can have many
  more).
- Awareness note: this verifier sees ONE plan, not the
  cross-suite consolidation. It does not need to dedupe
  cross-entry — that's the consolidation pass's job. The
  verifier's job is correctness of THIS plan.

## Telemetry

End-of-command summary gains:

- `evals/_judges.ts written: <N> canonical judges (deduped from
  <M> declarations)`.
- `evals/fixtures/: <K> case fixtures written`.
- Naming conflicts (if any): `judge "Foo" had divergent criteria
  in entries [a, b]; kept first`.

## Migration

- **Generator-side**: prompt updates emit `fixture` going
  forward. `setup` strings still parse for compat.
- **Renderer-side**: a plan with `fixture` writes the tree and
  emits `useFixture`; a plan with only `setup` falls back to the
  inline `harness.setup(script)` path (one-line bash heredoc).
  The fallback exists for plans hand-authored against the old
  contract; it is not produced by generation.
- **Existing on-disk eval files**: untouched. They continue to
  load via the data-array describeEval compat path. Authors
  regenerating a behavior get the new layout for that behavior;
  unrelated behaviors stay the way they were.

## Risks

- **Same-name divergent criteria.** Two entries declare
  `IdentifiesTriggerJudge` with subtly different criteria; we
  keep the first and log. Mitigation: the generator prompt
  pushes toward stable, scoped names so divergent
  same-named-judges should be rare. If they happen, the user
  audits via the warn event and renames one of the behaviors'
  judges in the spec/regen flow.
- **Fixtures-on-disk noise in PRs.** A 30-behavior skill ships
  ~50 small YAML files under `evals/fixtures/`. The PR diff is
  larger but each file is editable. Acceptable.
- **`_judges.ts` is a regenerated artifact.** Hand edits get
  clobbered. The file's banner says so loudly; we treat it the
  same way we treat spec.yaml's `# Managed by skillet` banner.
