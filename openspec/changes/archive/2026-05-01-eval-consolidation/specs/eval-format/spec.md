# Eval Format — Shared Judges + Disk Fixtures

## ADDED Requirements

### Requirement: Suite-wide `evals/_judges.ts`

Generated eval suites SHALL ship a single `evals/_judges.ts`
containing the canonical deduped set of named judges across all
behaviors in the skill. Each `.eval.ts` SHALL import the judges
it references from `./_judges.js`; per-file inline judge
declarations are NOT produced by generation.

The file SHALL carry a `Generated by skillet` banner indicating
that hand edits are clobbered on regen and that judge criteria
should be tuned via behavior-level changes that flow through the
consolidation pass.

#### Scenario: Eval file imports rather than inline-declares
- **WHEN** an eval is regenerated with the new flow
- **THEN** its `.eval.ts` contains an
  `import { ... } from "./_judges.js"` line listing the judges
  it uses
- **AND** it does NOT contain any `const FooJudge = judge(...)`
  declaration

#### Scenario: `_judges.ts` is regenerated alongside any eval
- **WHEN** any single behavior is regenerated
- **THEN** `_judges.ts` is rewritten with the consolidated set
  spanning the whole skill (not just that behavior's judges)

### Requirement: Disk-backed fixtures under `evals/fixtures/<case-slug>/`

Skillet SHALL write per-case fixture trees to
`evals/fixtures/<case-slug>/` whenever a generated case carries
a `fixture` field (a map of relative workspace paths to file
content), preserving the relative directory structure. The
generated `.eval.ts` SHALL call
`await harness.useFixture(<case-slug>)` instead of embedding the
fixture content as a shell-script string.

#### Scenario: Multi-file fixture written to disk
- **GIVEN** a case named `report-pwn-request__pr-target` with
  `fixture: { ".github/workflows/ci.yml": "...", "scripts/run.sh": "..." }`
- **WHEN** consolidation runs
- **THEN** `evals/fixtures/report-pwn-request__pr-target/.github/workflows/ci.yml`
  exists with the YAML content
- **AND** `evals/fixtures/report-pwn-request__pr-target/scripts/run.sh`
  exists with the script content
- **AND** the rendered eval body calls
  `await harness.useFixture("report-pwn-request__pr-target")`

#### Scenario: Cases without fixtures don't write a directory
- **GIVEN** a case with no `fixture` (and no legacy `setup`)
- **THEN** no entry under `evals/fixtures/` is created for it
- **AND** the rendered eval body has no setup call

### Requirement: `harness.useFixture(slug)` method

The harness type SHALL expose a
`useFixture(slug: string): Promise<void>` method that copies the
named fixture tree from `<skill-root>/evals/fixtures/<slug>/`
into the per-test workspace. Calling `useFixture` with a slug
that does not resolve to a directory SHALL throw a clear error
naming the missing path so authors find typos early.

#### Scenario: Missing fixture surfaces a clear error
- **WHEN** a generated `.eval.ts` calls
  `await harness.useFixture("nonexistent-slug")`
- **AND** `evals/fixtures/nonexistent-slug/` does not exist
- **THEN** the test fails with an error message naming the slug
  AND the resolved path that was searched

#### Scenario: Fixture tree mirrors workspace layout
- **WHEN** a fixture is laid out as
  `fixtures/<slug>/.github/workflows/ci.yml`
- **AND** the test calls `await harness.useFixture(<slug>)`
  followed by `await run("audit ...")`
- **THEN** the agent observes `.github/workflows/ci.yml` at the
  workspace root with that content

## REMOVED Requirements

### Requirement: Per-case workspace setup via harness helper

**Reason**: superseded by disk-backed fixtures + `useFixture`.
Inline-shell setup remains accepted as a backward-compat
fallback when a hand-authored plan lacks a `fixture` map, but
new generation does not produce it.

**Migration**: existing `.eval.ts` files using
`await harness.setup("…heredoc…")` continue to load and run.
Regenerating a behavior moves it to the new `useFixture` shape;
unrelated behaviors keep working as-is.
