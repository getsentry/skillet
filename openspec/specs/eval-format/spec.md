# Eval Format Specification

## Purpose

Eval files define test cases for agent skills. They are YAML files that live in a skill's `evals/` directory. Each file contains one or more eval cases specifying a workspace setup, user prompts, structural checks, and optional LLM judge criteria. The format is designed to be generated by agents and interpreted by the skillkit runner — no imports, no transpilation, no external dependencies.

## Requirements

### Requirement: File Discovery

The system SHALL discover eval files by globbing `evals/**/*.eval.yaml` relative to the skill root (the directory containing `SKILL.md`).

#### Scenario: Standard eval directory
- GIVEN a skill at `my-skill/` with `my-skill/evals/basic.eval.yaml`
- WHEN `skillkit eval my-skill/` runs
- THEN `basic.eval.yaml` is discovered and its cases executed

#### Scenario: Nested eval directories
- GIVEN eval files at `evals/core/format.eval.yaml` and `evals/edge/timeout.eval.yaml`
- WHEN `skillkit eval` runs
- THEN both files are discovered

#### Scenario: No eval files
- GIVEN a skill with no `evals/` directory or no `.eval.yaml` files
- WHEN `skillkit eval` runs
- THEN the system reports no evals found and exits with code 0

### Requirement: YAML Structure

Each `.eval.yaml` file SHALL contain a top-level `evals` array. Each entry in the array is one eval case.

#### Scenario: Minimal eval case
- GIVEN a YAML file containing:
  ```yaml
  evals:
    - name: "basic test"
      turns:
        - "do something"
  ```
- WHEN the file is parsed
- THEN one eval case is created with name "basic test" and one turn

#### Scenario: Full eval case
- GIVEN a YAML file containing all supported fields (name, workspace, turns, checks, criteria, threshold, timeout, requires)
- WHEN the file is parsed
- THEN all fields are correctly mapped to the eval case

### Requirement: Eval Case Fields

Each eval case SHALL support the following fields.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | YES | — | Human-readable name for the eval case |
| `workspace` | object | NO | empty tmpdir | Workspace configuration |
| `turns` | string[] | YES | — | User prompts sent to the agent sequentially |
| `checks` | array | NO | — | Structural assertions on workspace state and agent output |
| `criteria` | string | NO | — | Natural language criteria for LLM judge evaluation |
| `threshold` | number | NO | 0.75 | Minimum LLM judge score (0.0-1.0) to pass |
| `timeout` | number | NO | 120000 | Maximum milliseconds for the agent to complete all turns |
| `requires` | array | NO | — | Prerequisites; case is skipped if any are missing |

#### Scenario: Case with only structural checks
- GIVEN a case with `checks` but no `criteria`
- WHEN the case runs and all checks pass
- THEN the case passes (no LLM judge invoked)

#### Scenario: Case with only LLM criteria
- GIVEN a case with `criteria` but no `checks`
- WHEN the case runs and the judge scores above threshold
- THEN the case passes

#### Scenario: Case with both checks and criteria
- GIVEN a case with both `checks` and `criteria`
- WHEN the case runs
- THEN structural checks run first
- AND if any check fails, the case fails immediately (judge is not invoked)
- AND if all checks pass, the LLM judge evaluates criteria

### Requirement: Check Types

Checks SHALL support two categories: workspace checks (shell commands) and output checks (agent text).

#### Scenario: Shell command with regex match
- GIVEN a check `{ run: "git log -1 --format=%s", matches: "^feat\\(" }`
- WHEN the shell command executes in the workspace
- THEN the stdout is matched against the regex pattern
- AND the check passes if the regex matches

#### Scenario: Shell command with substring contains
- GIVEN a check `{ run: "cat output.txt", contains: "success" }`
- WHEN the shell command executes
- THEN the check passes if stdout contains the substring

#### Scenario: Shell command with not_contains
- GIVEN a check `{ run: "git log -1 --format=%B", not_contains: "\\n" }`
- WHEN the shell command executes
- THEN the check passes if stdout does NOT contain the substring

#### Scenario: Shell command with exact equals
- GIVEN a check `{ run: "git branch --show-current", equals: "main" }`
- WHEN the shell command executes
- THEN stdout is trimmed and compared exactly

#### Scenario: Shell command with exit code check
- GIVEN a check `{ run: "test -f AGENTS.md", exits: 0 }`
- WHEN the shell command executes
- THEN the check passes if the exit code matches

#### Scenario: Agent output contains check
- GIVEN a check `{ output_contains: "vulnerability" }`
- WHEN evaluated against the agent's concatenated text output from all turns
- THEN the check passes if the output contains the substring

#### Scenario: Agent output not_contains check
- GIVEN a check `{ output_not_contains: "credential escalation" }`
- WHEN evaluated against the agent's output
- THEN the check passes if the output does NOT contain the substring

#### Scenario: Agent output regex match
- GIVEN a check `{ output_matches: "line\\s+\\d+" }`
- WHEN evaluated against the agent's output
- THEN the check passes if the regex matches anywhere in the output

### Requirement: Requirements / Skip Conditions

Eval cases SHALL support `requires` to declare prerequisites. Cases are skipped (not failed) when prerequisites are missing.

#### Scenario: Required environment variable missing
- GIVEN a case with `requires: [{ env: "SENTRY_REPO" }]`
- WHEN `SENTRY_REPO` is not set
- THEN the case is skipped with reason "missing env: SENTRY_REPO"

#### Scenario: Required CLI tool missing
- GIVEN a case with `requires: [{ command: "gh" }]`
- WHEN `gh` is not found in PATH
- THEN the case is skipped with reason "missing command: gh"

#### Scenario: All requirements met
- GIVEN a case with `requires: [{ env: "HOME" }, { command: "git" }]`
- WHEN both are available
- THEN the case runs normally
